# Оптимізація збірки та бандла

Документ фіксує **поточний стан** production-збірки та **практичні напрями** покращення без обов’язкових змін у коді. Оновлюйте числові оцінки після `npm run build`, якщо змінюється залежність або конфіг Vite.

## Поточна картина (типовий `vite build`)

Орієнтовні розміри **після мініфікації** (фактичні імена чанків з хешем змінюються):

| Актив | Приблизний розмір (raw) | Примітка |
|--------|-------------------------|----------|
| Головний бандл (`index-*.js`) | ~357 KB | React, Zustand, Canvas, UI |
| **`StageView3D-*.js`** | **~919 KB** | Three.js + R3F + Drei + сцена |
| `exportBriefingPdf-*.js` | ~456 KB | jsPDF, autotable — **вже** окремий chunk (`import()` при експорті) |
| `html2canvas-*.js` | ~200 KB | Залежність PDF-шляху |
| `index.es-*.js` (jspdf) | ~151 KB | Підтягується при експорті |
| Шрифти Roboto (TTF) | ~2 × 515 KB | У PDF для кирилиці |
| CSS | ~38 KB | |

Vite попереджає про чанки **> 500 KB** — це **очікувано** для Three.js; збірка не вважається помилковою.

## Що вже зроблено добре

1. **Lazy-завантаження 3D** — `StageView3D` підвантажується через `React.lazy` лише в режимі 3D (`App.tsx`), головний бандл не містить увесь Three.js до першого відкриття 3D.
2. **Динамічний PDF** — `exportBriefingPdf` і важкі залежності не входять у критичний шлях першого екрану.
3. **PWA / Workbox** — кешування статики; `workbox` окремим малим чанком.

## Рекомендації (за пріоритетом зусиль / ефекту)

### 1. Політика попередження Vite (швидко)

Якщо попередження про 500 KB лише «шумить» у CI-логах, можна підняти `build.chunkSizeWarningLimit` у `vite.config.ts` (наприклад 900–1000) **лише** після явної згоди команди — щоб не приховувати реальне роздування майбутніх змін.

### 2. Подальший спліт усередині 3D (середня складність)

- Винести рідкісні меші / підкомпоненти в динамічні `import()` всередині `StageView3D.tsx` має сенс лише якщо профілювання покаже вигоду; R3F уже дерев’янить багато що.
- **Draco / менші геометрії** — для MVP з примітивів вигода мала.

### 3. Three.js «на дієті» (висока складність)

Повна заміна на custom build або `three-stdlib` лише для частини API — дорого в підтримці. Розглядати лише за жорстких обмежень мобільного трафіку та метрик LCP.

### 4. Шрифти PDF

Roboto підключені для якості PDF; **підміна на системні шрифти** у jsPDF зменшить precache, але погіршить передбачуваність верстки кирилиці — продуктове рішення.

### 5. Візуальний аналіз бандла (за потреби)

Опційно один раз додати **`rollup-plugin-visualizer`** (або аналог) у `vite.config.ts`, зібрати, подивитись treemap, потім **прибрати** плагін з постійного конфігу — щоб не ускладнювати CI.

```bash
npm run build
# після додавання visualizer — відкрити згенерований звіт у браузері
```

## Тести як запобігання регресіям перформансу

Логіка **view transform**, **A4/PDF розміри** та **поле / reclamp** покриті unit-тестами (`src/**/*.test.ts`) — зміни в цих модулях не повинні ламати геометрію плану та експорт «тихо».
