Чернетка для узгодження перед реалізацією. Межі та формулювання можна змінити після review.

## 1. Цілі

| Тема | Результат для користувача |
|------|---------------------------|
| **Розмір площадки** | Задати **довільні** ширина × довжина в метрах у межах продукту, без прив’язки лише до пресетів. |
| **Обмеження** | Верхні межі: **ширина до 50 м**, **довжина до 100 м** (відповідає осям `fieldSizeM.x` / `fieldSizeM.y` у коді; якщо в UI трактувати інакше — узгодити перед кодом). |
| **Сітка** | **Шахматка** з кроком **1 м** на 2D-плані (читабельність на великих полях). |
| **3D для PDF** | Окремий режим перегляду 3D, узгоджений з тим, що потрапляє в PNG для PDF (співвідношення сторін, кадр). |
| **Брифінг** | Коротка **підказка** (текст), що пояснює: як отримати коректний знімок для PDF / увімкнути режим «як у PDF». |
| **Responsive (R)** | Картка перегляду 2D/3D **розтягується** в доступній площі вікна на великих екранах, без зайвих порожніх полів по боках/знизу. |

## 2. Залежності між блоками

```
[R] Responsive layout (картка сцени — flex/max-width breakpoints)
         │
         │ (необов’язкова залежність: покращує тестування A і Б)
         ▼
[А] Роздільні max для ширини/довжини поля + UI + clamp + JSON
         │
         ├──────────────────────────────────────┐
         ▼                                      ▼
[Б] Сітка 1 м шахматка (2D)              [В] 3D «PDF preview»
         │                                      │
         │                                      ├─ спільне з [Г] співвідношення знімка
         ▼                                      ▼
[Г] Підказка в брифі ◄────────────────── однакова термінологія з UI
```

- **R** не залежить від **А**; можна змінювати порядок релізів, але **черга 1→2** вище — рекомендована.
- **А** краще зробити першим серед «даних сцени» або паралельно з **R** з обережністю до продуктивності 2D на 100×50 м.
- **В** і **Г** логічно прив’язати до вже існуючих `pdfSnapshotPixelSize` / `stageViewportAspectRatio` (`a4PrintLayout.ts`, `StageView3D` capture).
- **Б** візуальна зміна: може не чіпати крок **snap** (0,5 м), якщо домовимось лише про **малювання** шахматниці по 1 м.

## 3. Фази робіт (рекомендований порядок)

### Фаза R — Responsive картка 2D/3D (великі екрани)

1. **Розмітка:** центральна колонка з планом/3D — `flex: 1`, `min-height: 0`, щоб блок **заповнював** висоту між шапкою/брифінгом (уточнити поточну структуру `App.tsx` / CSS).
2. **Ширина:** на великих breakpoints (`min-width: …`) підняти або прибрати **max-width** у контейнера картки, щоб не лишались широкі «поля» порожнього простору (або залишити обмеження лише для тексту брифінгу).
3. **Перевірка:** вузький вікно / мобільний layout **не** ламається (існуючі breakpoints).
4. **Зв’язок з планом:** це **не** змінює метри поля; знімок PDF не залежить від розміру вікна браузера.

**Ризики:** регресій у мобільному layout; перевірити `StageCanvas` resize / `ResizeObserver`.

---

### Фаза A — Довільний розмір поля + асиметричні ліміти

1. **Домен** (`field.ts`): замінити єдиний `maxM` на щось на кшталт `maxWidthM: 50`, `maxHeightM: 100` (назви узгодити з UX).
2. **`clampFieldDimensions`**: окремо обмежувати `x` і `y`.
3. **UI** (`App.tsx` / тулбар): поля вводу або один діалог «Ширина × Довжина» з валідацією; пресети залишити як швидкі кнопки, додати 1–2 пресети під довгі дистанції (за потреби).
4. **`*.stage.json`**: формат уже зберігає два числа — міграція схеми зазвичай не потрібна; перевірити імпорт старих файлів.
5. **Тести**: граничні значення, зміна розміру з уже розставленими об’єктами (існуюча логіка clamp об’єктів).

**Ризики:** продуктивність 2D-сітки та міні-карти на 50×100 м — можливі оптимізації `drawGrid` (рідший малюнок дрібних ліній при малому масштабі), окремий пункт у технічному спринті.

---

### Фаза B — Шахматка 1 м на 2D

1. У `drawGrid` (або окремий шар) малювати **заливку клітин** кроком **1 м** у два кольори (наприклад, `(i+j) % 2`).
2. Залишити поточну сітку **0,5 м** як тонкі лінії **або** перемкнути основні лінії на 1 м і дрібні 0,5 м лише при високому zoom — **рішення UX** (підказка в плані: спочатку мінімальний варіант — шахматка поверх існуючої сітки).
3. **Доступність:** достатній контраст у світлій темі.

**Залежність від А:** на дуже великому полі шахматка з 2500 клітин за поганого zoom може бути важкою — варто обмежити малювання видимим viewport (як для інших елементів).

**Реалізація (зроблено):** `GRID_CHESS_M` у `field.ts`; `drawChessboard1m` у `StageCanvas.tsx` перед лініями `GRID_SNAP_M`; пропуск при малій клітині на екрані або при кількості клітин у кадрі понад 8000.

---

### Фаза C — Режим 3D «Для друку в PDF»

1. **Камера / viewport:** те саме **aspect ratio**, що й `pdfSnapshotPixelSize(fw, fh)` (вже є прив’язка до поля).
2. **Поведінка:** перемикач поруч із «Огляд / Зона стрільця»; у режимі PDF — опційно **обмежити** або **підказати** орбіту так, щоб кадр відповідав знімку; за бажання — легка **рамка safe area** на екрані.
3. **Зв’язок із експортом:** при натисканні «Завантажити PDF» перевіряти, чи користувач був у 3D і в цьому режимі (м’яке попередження, не блокер — як домовимось).
4. **Технічно:** спільна функція «цільовий aspect для PDF» з `StageView3D` і `exportBriefingPdf` / capture, щоб не роз’їхались.

---

### Фаза D — Підказка в брифі

1. Додати поле або **статичний інформаційний блок** під таблицею брифінгу (не обов’язково в JSON): 2–3 речення UK/EN.
2. Текст узгодити з командою (приклад змісту): *«Для знімка в PDF відкрийте 3D, увімкніть режим „Як у PDF“, перевірте кадр, потім „Завантажити PDF“»*.
3. Потрапляння в експортований **PDF** — за бажанням (другий абзац у дусі «друкована копія містить підказку») або лише в веб-UI.

## 4. Критерії готовності (коротко)

- **R:** на широкому десктопі картка 2D/3D займає помітно більше площі вікна, ніж зараз, без поламаного мобільного вигляду.
- Поле можна задати вручну в межах 50×100 м (за узгодженими осями), файли вправи відкриваються назад.
- На 2D видно шахматку 1 м без критичних лагів на типовому ПК.
- Є режим 3D, який візуально відповідає кадру PDF (aspect), з описом у UI.
- У брифінгу є зрозуміла підказка про послідовність дій для PDF.

---

## 5. Відкриті питання

1. **50 м і 100 м:** чітко зафіксувати: `x ≤ 50`, `y ≤ 100` чи навпаки (залежно від того, як у вас названі «ширина» і «довжина» на плані).
2. **Snap:** лишаємо 0,5 м для розстановки чи додаємо опцію snap 1 м?
3. **Шахматка:** тільки 2D чи потрібен натяк на підлогу в 3D (зазвичай ні на першому етапі).
4. **PDF-підказка:** лише в застосунку чи також у згенерованому PDF?
