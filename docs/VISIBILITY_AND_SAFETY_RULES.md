# Правила видимості та безпеки (чернетка для BL-010 / BL-013)

Документ описує **як планується** реалізувати аналіз «чи видно мішень з точки» і «чи перекриває реквізит промінь» у **2D-проєкції плану** (MVP). Це основа для **BL-010** (кут 180°, зона штрафних ліній) і **BL-013** (аналіз прострілів для дизайнеру). Деталі домену мішеней і реквізиту — [`src/domain/models.ts`](../src/domain/models.ts).

---

## 1. Мета та межі MVP

| Аспект | Рішення на перший етап |
|--------|-------------------------|
| Вимір | **План 2D**, метри; вісі як у редакторі (X/Y поля). |
| Висота 3D | **Не моделюємо** в першому проході: реквізит задає **перешкоду в площині плану** за **проєкцією лиця / тіла** (прямокутник, відрізок, полігон). Фраза «щити необмежені вгору» в реальному світі означає: **вертикальна площина щита достатньо висока**, щоб перекривати огляд — у **top-down** 2D це еквівалентно тому, що **блокуємо промінь**, якщо він перетинає **проєкцію лиця щита** на план (прямокутник за `position`, `sizeM`, `rotationRad`). Окремо нижче — винятки для «тонких» об’єктів. |
| Точка стрільця | Дискретний набір точок (сітка або точки вздовж допустимої зони відносно **штрафних ліній** — логіка вибору точок для BL-010 ще в `stageStore` / окремому модулі). |
| Мішень | Точка **центру** мішені (`Target.position`) або малий круг/квадрат габариту — узгодити з тим, як уже малюється план. |
| Результат | Для пари (точка, мішень): **видимо / не видимо** і опційно **id першої перешкоди** на промені. |

**Що не входить у MVP:** кулькова траєкторія, нахил стрільця, динамічна висота мішені окремо від «точки прицілювання».

## 2. Геометрична модель

1. **Промінь** — відрізок від точки **S** (стрілок) до точки **T** (центр мішені) або напівпряма **S → T** для перевірки перетину.
2. **Перешкода** — у 2D одне з:
   - **Відрізок** (тонка штрафна лінія, край порту — якщо спрощуємо).
   - **Прямокутник** (щит, двері, стіл, бочка як квадрат у плані тощо) — обертання `rotationRad`.
   - **Полігон** (складніші щити з портом) — краще **другий етап**: спочатку **bounding box** або **набір відрізків** з існуючого коду малювання плану.

3. **Перетин променя з перешкодою:** стандартні алгоритми: промінь–відрізок (segment), промінь–прямокутник (чотири відрізки або SAT).

4. **Порядок:** сортувати перешкоди за **відстанню від S** вздовж променя; перша перешкода, що перетинається, **блокує** видимість (якщо не рахуємо «простріл крізь отвір» — окреме правило для портів).

## 3. Правила за типом реквізиту (`PropType`)

Узгоджено з коментарями в `models.ts`. Для кожного типу: **форма в 2D**, **чи блокує промінь**, **примітки**.

| Тип | Проєкція на план (MVP) | Блокує промінь? | Примітки |
|-----|-------------------------|-------------------|----------|
| `faultLine` | Відрізок довжиною `sizeM.x`, товщина мінімальна (5 см у домені — можна як **лінія** без товщини або з epsilon). | **Так**, якщо промінь перетинає відрізок. | Штрафна лінія задає **межу зони**; **цільова модель** — **BL-019** (розділ 4): **об’єднання** відокремлених контурів; **отвори**; старт **0/1/кілька** або лише брифінг «будь-де». Поки окремі відрізки — зона наближена. Заливка штрафної зони — п. **4.5**; лінії **під** реквізитом — п. **4.6**; UX незамкненого контуру — п. **4.3**. |
| `shield`, `shieldDouble`, `shieldWithPort`, `shieldPortLow/High/Slanted`, `shieldWithPortDoor` | Прямокутник **лиця**: `sizeM.x` × товщина `sizeM.y` (~0,05 м), `rotationRad`. | **Так** (непрозоре лице). | «Необмежене вгору» в 3D у 2D top-down уже покривається **повною шириною лиця** в плані; без висоти в MVP **не** додаємо додатковий «високий» блок — за потреби пізніше: розширення полігону. |
| `door` | Аналогічно щиту (1×0,05 на плані). | **Так**. | |
| `barrel`, `tireStack` | Квадрат/прямокутник `sizeM` (типово ~0,62×0,62). | **Так**. | |
| `woodTable`, `woodChair` | Прямокутник опори. | **Так** (спрощено все тіло). | |
| `weaponRackPyramid` | Прямокутник у плані з `propGeometry`. | **Так** (спрощено). | |
| `seesaw`, `movingPlatform`, `cooperTunnel` | Полігон або прямокутник з існуючими константами в `propGeometry.ts`. | **Так** / частково | Потрібні **точні контури** з коду малювання 2D — не вигадувати з нуля. |
| `startPosition` | Не перешкода для променя до мішені (або **не враховуємо** як блок). | **Ні** (за замовчуванням). | |

**Мішені:** для променя **S → T** мішень можна трактувати як **точку**; для перевірки «чи мішень у секторі 180°» використовується **вектор напрямку стрільби** з точки S (узгодити з брифінгом / локальною віссю «вниз по полю»).

## 4. Зона всередині штрафних ліній (логіка для обговорення)

Цей блок — **не** поточний код; це узгоджений напрямок, щоб **чітко** визначити допустиму зону для BL-010 і не лишати «дірки» між сегментами.

### 4.1 Як зараз (спрощено)

- Кожна **штрафна лінія** — окремий елемент `faultLine` у сцені: позиція, обертання, довжина (`sizeM.x`) — див. `faultLineEndPointsWorld` у `propGeometry.ts`, малювання в `StageCanvas.tsx`.
- Кілька **незалежних** відрізків **рідко** без зазорів утворюють **замкнутий** контур; на практиці зона «в межах штрафних ліній» залишається **неоднозначною** (отвори між кінцями сегментів).

### 4.2 Пропозиція: полілінія з послідовних точок

1. Користувач (або РО) ставить **точки по порядку** (вершини контуру).
2. Між **послідовними** точками **автоматично** домальовуються **сегменти штрафних ліній** (геометрія випливає з ланцюга точок).
3. **Замикання контуру:** остання вершина збігається з **першою** (або окрема дія «замкнути»), з **допуском epsilon = 0,05 м (5 см)** — однаково для зовнішнього контуру, **внутрішніх** контурів (дірок) і відокремлених полігонів. **Незамкнений** контур під час роботи **дозволений** (автор малює **частинами** і закриває контур згодом); деталі повідомлення й редагування вершин — п. **4.3**.
4. **Кілька окремих замкнених контурів** (полігони **не** вкладені одне в одне): **допустима зона стрільби** — **об’єднання** (union) інтер’єрів усіх таких полігонів. Стрільець може **вільно переміщуватися** між цими зонами — точка **S** може лежати в **будь-якій** компоненті цього об’єднання (узгоджено з текстом брифінгу та регламентом вправи).

5. **Стартові маркери на плані** (`startPosition`): на сцені може **не бути жодного**, бути **один** або **кілька**. Вони слугують **підказкою для брифінгу та візуалу** («звідки стартуємо»), а не обмеженням «рух лише від цього маркера»: при **кількох відокремлених** зонах кілька стартів можуть відповідати **різним** зонам, але **допустима геометрична область** для перевірок лишається **об’єднанням** штрафних контурів (п. 4), якщо вправою не задано вузькішу логіку в тексті брифінгу.

6. **Старту немає на плані:** допустимий сценарій — у брифінгу формулювання на кшталт **«старт будь-де в межах штрафних ліній»**. Тоді для геометричних перевірок (зона допуску, дискретизація точок **S**) область така сама: **усередині об’єднання** контурів з урахуванням **отворів** (п. 7–8); маркер старту **не** обов’язковий.

7. **«Матрьошка» (зовнішній контур + дірки / отвори):** є **зовнішня** замкнена лінія та один або кілька **внутрішніх** замкнених контурів усередині. **Штрафною зоною** в цьому узгодженні вважається область **всередині зовнішньої** межі **без** інтер’єрів дірок — тобто **кільце** (сума «рамок») між зовнішнім і внутрішніми лініями; **рух у цій червоній смузі** у моделі продукту — **поза** дозволеним острівцем усередині дірки. **Усередині дірки** — **не** ця штрафна смуга (п. 4.5 — заливка). **Правила малювання** для внутрішніх контурів — **ті самі**, що в п. **1–3** (полілінія з точок, сегменти, замикання з epsilon **5 см**, незамкненість і повідомлення з п. **4.3**): кожна дірка — **окремий** замкнений контур, заданий тим самим інструментом, що й зовнішня межа. **П. 5–6** (старт / брифінг) для сцени з матрьошкою застосовуються **без змін:** геометрично допустима зона стрільби для перевірок — **об’єднання** «островів» (інтер’єри дірок і, за наявності, відокремлених полігонів п. 4), **не** червоне кільце; маркери старту й текст брифінгу — як у п. 5–6.

8. **Відокремлені полігони без вкладеності** (п. 4): заливка й логіка **union** — окремо від п. 7; **неоднозначність «кільце vs дірка»** тут не виникає. Якщо для **складної** сцени з отворами **і** без старту на плані потрібен додатковий вибір області для автоматичного аналізу — **відкрито** (редактор / брифінг; для «матрьошки» візуально однозначно за п. 7).

9. **Геометрія в коді:** `polygon ∪ polygon` для відокремлених островів; для «матрьошки» — зовнішній полігон **мінус** union дірок (або еквівалент `point-in-polygon` з отворами); `isAllowedShootingPoint(S)` з урахуванням п. 4–8.

*Наслідок для продукту:* зміна **моделі даних** (ланцюги вершин, кілька полігонів, отвори) — **епік** + міграція; до цього BL-010 — лише **наближені** евристики.

### 4.3 Редактор: незамкнений контур і редагування вершин (узгоджено)

- **Незамкнений контур дозволено:** під час малювання автор може залишати контур **відкритим** (малювання **частинами**), щоб згодом **замкнути** його в цілому.
- **Повідомлення:** поки контур **не замкнений**, показувати стійке (або повторюване) повідомлення: **«Контур штрафної зони не замкнений»** — без блокування збереження чернетки сцени, якщо продуктово не вирішать інакше (аналітика зон BL-010 для незамкненого контуру — **не застосовувати** або евристика).
- **Редагування:** дозволити **перетягувати** вже проставлені **вершини** і за потреби **розмикати** контур (зняти збіг останньої й першої точки / відкрити полілінію), щоб зручно підлаштувати вправу. Після **розмикання** заливка зникає (див. п. **4.5**).

### 4.4 Міграція зі старих сцен

- **Перекреслення** користувачем у новому режимі; **автоматичне** «зшиття» старих окремих `faultLine` в один контур **не** обіцяємо як надійне.
- Поки контур **не замкнутий** — повідомлення як у п. **4.3**.

### 4.5 Візуалізація штрафної зони в редакторі (узгоджено)

- **Коли заливати:** червону (або узгоджену) заливку показувати **лише після замикання** контуру. Поки контур **відкритий** — лише **полілінія** (сегменти між точками), **без** заливки площі; це **візуально підказує** автору: з’явилася заливка — контур **замкнений**. Після **розмикання** заливку **прибирати**, доки контур знову не замкнуть.
- **«Матрьошка» (зовнішній контур + дірка):** дуже **легка напівпрозора червона** заливка для області **між** зовнішньою та внутрішніми лініями (**кільце**) — це **штрафна смуга**; **внутрішність дірки** — **без** червоного (так само **після** замикання відповідних контурів).
- **Острови / дірки / «не штрафна» площа** (інтер’єри відокремлених полігонів п. 4 і внутрішність дірок у матрьошці): заливати **як загальне покриття майданчика поза червоною штрафною смугою** — **пісок, земля або трава** залежно від обраного для сцени покриття поля (у продукті — той самий візуальний стиль, що й для площі вправи; у коді орієнтир **`fieldGroundCover3d`** / узгоджений 2D-фон плану). Так **відокремлені** зони стрільби й **острови** всередині матрьошки візуально узгоджені з полем, на відміну від червоного кільця.
- **Що малюється:** площина **між** точками сегментів (полігональна заливка) з урахуванням отворів; лінії штрафних лишаються поверх заливки.
- **Шари (напрямок):** заливка — **під** лініями штрафних, **над** сіткою / фоном поля; реквізит — за потреби **поверх** ліній (п. 4.6).

### 4.6 Щити та інший декор — **поза** полігоном; накладання на лінії

**Узгоджено:** щити **не** входять у визначення полігона штрафних ліній; додаються **окремо** як `props`, без прапорців «на межі».

**Візуалізація й редагування:** штрафні лінії можуть **проходити під** реквізитом (умовно **під** щитом, стосом шин, **бочкою, столом, стільцем**): ці об’єкти можна **розміщувати поверх** ліній на плані. Лінії **не** обрізаються по контуру реквізиту — це **шар малювання** (лінії «під» декором), а не логічне розбиття сегментів по перешкодах.

Для **видимості та прострілів** (розділи 2–3, BL-010 / BL-013) реквізит лишається **перешкодою для променя** окремо від межі зони; якщо щит **впритул** до межі — контекст зрозумілий **візуально**, без окремої формальної перевірки «дотику».

### 4.7 Зв’язок з розділами 2–3 і наступними фічами

Поки полігон штрафних ліній формується з **окремих** `faultLine`, для «зони всередині» потрібні **ефективні евристики** або обмежений MVP. Після впровадження **полілінії + замикання + об’єднання зон / отвори / опційні старти** (п. 4.2) логіка допустимої зони стає **формальною** — **база** для **перекриття променя** та **прострілів** (BL-010 / BL-013). Див. **[BACKLOG.md](./BACKLOG.md)** — **BL-019**, **BL-010**, **BL-013**.

### 4.8 Що лишається узгодити на рівні реалізації (не блокує старт BL-019)

**Скоп BL-019:** реалізувати **повний** обсяг продуктового опису в §4 **одним** етапом — **без** розбиття на вузький MVP (у т. ч. **кілька контурів**, **отвори / матрьошка**, **union**, редактор з незамкненим контуром і drag вершин, заливки червоним кільцем і покриттям поля на островах, **epsilon 5 см**, міграція зі старих `faultLine` за правилами §4.4).

| Питання | Примітка |
|--------|----------|
| **Формат `*.stage.json`** | Структура масивів вершин / контурів — описати в **[TECH.md](./TECH.md)** при впровадженні + за потреби підняти `STAGE_PROJECT_VERSION`. |
| **i18n** | Рядок **«Контур штрафної зони не замкнений»** — ключі UK/EN у `messages.ts`. |

Продуктові правила з §4 для **epsilon**, **матрьошки**, **заливок** (червоний кільце / покриття поля на островах) — **узгоджені** з поточним обговоренням.

## 5. Реалізація в коді (план)

### 5.1. Шар `src/domain/`

1. **`visibility2d.ts`** (назва умовна):
   - `segmentIntersectRay(seg, ray)` / `segmentIntersectionParameter(...)` — чисті функції, без React.
   - `rectOutlineAsSegments(obb)` — прямокутник з центру, розміру, кута → 4 відрізки.
   - `firstBlockingPropAlongRay(s, t, props, rules): Prop | null` — перебір `props` з відсівом за типом; повертає першу перешкоду за `t` ∈ (0,1] на відрізку S–T.
   - `isTargetVisibleFromPoint(s, targetCenter, props, ...): boolean`.

2. **`visibilityRules.ts`** (або константа `PROP_BLOCKS_RAY: Record<PropType, boolean>`):
   - Мапінг типу → **геометрія перешкоди** (фабрика або switch).

3. **Тести** `visibility2d.test.ts`: відомі конфігурації «промінь перетинає / не перетинає» щит.

### 5.2. Інтеграція (пізніше)

- **BL-010:** після обчислення видимості — перевірка **куту 180°** відносно напрямку безпеки; агрегація попереджень у `StageCanvas` або окремий хук.
- **BL-013:** візуалізація «зон видимості» — можливо **кеш** сітки або вибірки точок; окремий режим UI.

### 5.3. Залежності від існуючого коду

- Координати та **обертання** реквізиту вже в `Prop` — узгодити з тим, як **draw** у `StageCanvas` / `propGeometry` для **контурів** щитів з портом (якщо прямокутник недостатній — етап 2: полігон з тих самих функцій, що й малювання).

## 6. Що ще треба врахувати (обговорення)

1. **Кут 180°:** відносно **якого вектора** на кожній точці S — глобальний «downrange» (наприклад +Y поля), локальний **нормаль до штрафних ліній**, чи напрямок до **найближчої** мішені? Це впливає на формулу «мішень за межами півкола».
2. **Штрафні лінії / зона:** див. **розділ 4** — **об’єднання** відокремлених зон, **отвори**, старт **опційний**; до впровадження — евристики або обмежений MVP.
3. **Порти в щитах:** промінь через **отвір** — у MVP можна **ігнорувати** отвір (щит суцільний) або **другий етап**: два прямокутники ліва/права частина лиця.
4. **Продуктивність:** O(точки × мішені × реквізит)** — для великої сцени потрібен **spatial index** (простий grid по полю) або зменшення точок.
5. **Юридичне застереження:** у UI залишити формулювання **підказка**, не офіційна перевірка безпеки.
