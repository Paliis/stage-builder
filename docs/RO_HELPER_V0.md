# RO Helper v0 — повна специфікація контенту та продукту

**Статус:** специфікація для розробки модуля на порталі **Shooters Tools** (`/ro-helper`). Код модуля може ще не існувати; цей документ — джерело істини для контенту, UX і технічної інтеграції.

**Зв’язок:** [PORTAL_PLAN.md](./PORTAL_PLAN.md) (модулі, дисклеймер, roadmap), [TECH.md](./TECH.md) (PWA, аналітика, збірка).

**Останнє оновлення:** квітень 2026 (двомовність UK+EN, дисципліни IPSC, відповідальність ФПСУ).

---

## 1. Мета v0, аудиторія, не-цілі

**Мета:** швидкий довідник для **RO** на лінії та під час підготовки: безпека, процедури, скоринг, спорядження, адміністрування матчу — з **посиланнями на первинні правила** і **дисклеймером**.

**Аудиторія:** Range Officers, помічники, дизайнери/MD (орієнтир), стрільці; у т. ч. **іноземні** учасники та RO, яким потрібна **англійська** версія того самого матеріалу.

**Двомовність (обов’язково для релізу статті):** для кожного `slug` існують **дві** версії контенту — **`uk`** і **`en`** (див. [§5.1](#51-двомовність-uk--en-та-шари-ipsc--fpsu)). Публікація лише однією мовою без пари — **неповний** реліз; допускається черга `draft`, але `published` лише коли обидві пройшли рев’ю.

**Не-цілі v0:**

- Повне дзеркало rulebook IPSC у застосунку.
- Юридично значуща заміна **Range Master**, арбітражу або документів конкретного матчу.
- AI-асистент з тлумаченням правил (пізніше, окремий епік).
- Вичерпне покриття кожного підпункту rulebook без пропусків (залишаємо довідником по темах, не «PDF у кишені»).

## 2. Ролі та контентний пайплайн (модель C)

**Погоджено:** чернетки текстів готує **LLM / асистент**; **власник продукту** не пише весь обсяг вручну, але **обов’язково рев’ює та затверджує** кожну статтю перед публікацією.

**Заборонено:** публікація статті без людського рев’ю.

### 2.1. Чекліст рев’ю (перед релізом статті)

1. Кожен зазначений **номер пункту IPSC** звірено з **офіційним PDF/сторінкою** обраної редакції (не «з пам’яті моделі»).
2. У frontmatter вказано **`ipsc_edition`** (рік або ідентифікатор редакції) і **`primary_url`** (посилання на первинник).
3. Блок **«Локально (ФПСУ)»** (якщо є) має **джерело** (URL або назва документа ФПСУ) і не суперечить дисклеймеру.
4. Немає **вигаданих** цитат довжиною як офіційний текст; у тілі — переважно **підсумок** мовою статті + короткі витяги згідно з [§10](#10-quick-cite-та-авторське-право).
5. Скоринг і числа (**Major/Minor**, папір/метал) — **лише після звірки** з таблицями обраної редакції **для обраної дисципліни**; при сумніві — загальна відправка до rulebook без конкретних цифр.
6. За можливості **друга думка** (досвідчений RO) на статті категорії **Safety / DQ** — вибірково.
7. Для пари **UK + EN**: зміст **відповідає** одна одній (ті самі пункти IPSC, той самий шар ФПСУ); відмінності формулювань через мову, не через правила.
8. Блок **«Локально (ФПСУ)»**: див. [§4.1](#41-хто-фіксує-відмінності-фпсу-від-ipsc) — **не** додавати без підтвердження джерелом і рев’юера.

### 2.2. Поля контенту (рекомендований frontmatter)

```yaml
title: "…"                 # мова = locale
slug: "kebab-case-en"      # стабільний URL, латиниця; однаковий для uk/en пари
category: safety|penalties|scoring|equipment|match-admin
locale: uk|en
discipline: handgun|pcc|rifle|mini_rifle|shotgun  # жодна дисципліна не «канон» для всього модуля; primary_url = PDF/HTML саме цієї дисципліни
ipsc_edition: "YYYY або label"   # заповнити при релізі
ipsc_refs:
  - rule: "10.5.2"
    note: "опційно"
primary_url: "https://…"   # офіційний PDF/HTML IPSC для цієї дисципліни
fpsu_refs: []              # опційно: локальні документи (URL / назва)
fpsu_delta_verified: false # true лише після п. §4.1
reviewer: "ім’я"           # хто підписав рев’ю цієї мовної версії
review_date: "YYYY-MM-DD"
draft_source: "llm"        # опційно
status: draft|reviewed|published
```

Пара UK/EN: два файли з однаковим `slug` у каталогах `uk/` та `en/` (див. [RO_HELPER_CONTENT_TZ.md](./RO_HELPER_CONTENT_TZ.md)).

## 3. Юридичний статус і дисклеймер (обов’язково в UI)

Текст для модуля (українською; коротка версія):

- **Shooters Tools / RO Helper** не є офіційним джерелом правил **IPSC** чи національних федерацій.
- Матеріали **не замінюють** рішення **Range Master**, арбітражу на матчі або **Match Book** конкретного змагання.
- Цитати та нумерація пунктів мають **посилання на первинник** і **вказану редакцію** правил IPSC; документи **ФПСУ** та інших федерацій можуть відрізнятися нумерацією та трактуваннями — на місці діють **офіційні документи організатора**.

Повніша юридична рамка узгоджена з [PORTAL_PLAN.md](./PORTAL_PLAN.md) §3.2–3.3.

## 4. Базис IPSC і локальні шари (ФПСУ та ін.)

- **IPSC** — канонічний **базис**: якір нумерації в статтях і в UI.
- **ФПСУ** (і згодом інші коди `local_*`) — **накладення**: лише де є відмінність, доповнення або інше трактування; не дублювати повний текст, якщо збігається з IPSC.

**UX (v0+):**

- **Базис:** IPSC (фіксовано в логіці продукту).
- **Перемикач:** «Показувати примітки **ФПСУ**» (увімкнено за замовчуванням для UA-аудиторії — за продуктовим рішенням).
- Збереження: **`localStorage`** (ключ на кшталт `ro-helper-layer-fpsu`).

**Маппінг пунктів:** лише **ручний** у frontmatter; автоматичне зіставлення редакцій **не** очікується в v0. Якщо немає 1:1 — окрема стаття з міткою **«Локально»** і посиланням на PDF ФПСУ.

### 4.1. Хто фіксує відмінності ФПСУ від IPSC

**Хто порівнює і «каже, що ФПСУ відрізняється»:** не LLM і не анонімне джерело.

- **Обов’язково:** **рев’юер** (власник продукту або явно призначена людина) порівнює **офіційний** документ ФПСУ з відповідним фрагментом **IPSC** (`primary_url` для тієї ж дисципліни) і **сам** формулює блок «Локально (ФПСУ)» або затверджує текст після AI-чернетки.
- **Опційно:** консультант з ФПСУ / досвідчений RO з роботи на національних матчах — для **підпису** або вибіркової перевірки; тоді в реєстрі варто поле `fpsu_consultant` (внутрішньо).
- **Якщо відмінність не підтверджена документом ФПСУ:** блок ФПСУ **не публікується**; залишається лише IPSC. Поле `fpsu_delta_verified: false` за замовчуванням; `true` лише після явної перевірки.
- **LLM:** може запропонувати **чернетку** порівняння лише якщо в промпт вкладено **цитати/витяги з обох** PDF і посилання; фінальне рішення «чи є відмінність» — за рев’юером.

## 5. Інформаційна архітектура та URL

- **Модуль:** стабільний шлях **`/ro-helper`** (див. [PORTAL_PLAN.md](./PORTAL_PLAN.md) §4.2).
- **Статті:** `/ro-helper/:category/:slug` (приклад: `/ro-helper/safety/break-180-handgun`). Мова контенту **не** в URL на v0 — узгоджується з **глобальною мовою UI** порталу (як зараз UK/EN у Stage Builder), див. [§5.1](#51-двомовність-uk--en-та-шари-ipsc--fpsu).
- **Головна модуля:** `/ro-helper` — великі категорійні плитки (SOS) + дисклеймер.
- **Портал:** окрема **картка** на головній [`/`](../src/portal/PortalHome.tsx) — після імплементації.

Категорії (`category`) і кольори — [розділ 7](#7-uxui-sos-плитки-та-кольори).

### 5.1. Двомовність (UK + EN) та шари IPSC / ФПСУ

**Два незалежні перемикачі для користувача:**

1. **Мова інтерфейсу та статті** — `uk` | `en` (спільно з порталом / `I18nProvider`): показувати тіло з `content/.../uk/` або `.../en/` для поточного `slug`.
2. **Шар правил** — **лише IPSC** | **IPSC + примітки ФПСУ** (зберігання в `localStorage`, див. [§4](#4-базис-ipsc-і-локальні-шари-фпсу-та-ін)).

Іноземні RO/стрільці обирають **EN** + за потреби вимикають ФПСУ, якщо працюють не в українському контексті.

**Контент:** кожен `slug` має бути реалізований **двічі** (UK і EN). Назви розділів у EN-файлі — англійською (наприклад `## What this is`, `## RO actions`); зміст узгоджений з UK-парою.

**Дисципліна:** у продукті передбачити вибір **виду стрільби** (`handgun` | `pcc` | `rifle` | `mini_rifle` | `shotgun`) так само, як мову та шар ФПСУ. Кожна стаття має **явний** `discipline` і **свій** `primary_url` на відповідний rulebook IPSC; пістолет — **один з п’яти рівноправних** варіантів, без привілею «дефолтного» slug чи URL.

## 6. Дисципліни (види стрільби) та документи IPSC

У **IPSC** для різних дисциплін діють **різні** офіційні документи. У кожній статті **`primary_url`** має вказувати на **той** rulebook (або сторінку завантаження), який відповідає полю **`discipline`**.

| `discipline` | Примітка для контенту |
|--------------|----------------------|
| `handgun` | Окремий PDF Handgun; пункти лише з нього |
| `pcc` | Окремий PDF PCC; не підміняти handgun |
| `rifle` | Окремий PDF Rifle |
| `mini_rifle` | Окремий PDF Mini Rifle |
| `shotgun` | Окремий PDF Shotgun |

**Контент-пакет v0:** теми з [§8](#8-повна-структура-контенту-категорії-та-статті-v0) реалізуються для **усіх п’яти** дисциплін одразу (див. інтро §8). Один і той самий «базовий» slug без суфікса **не** використовувати для різних `discipline` на одному шляху `/ro-helper/...` — для кожної дисципліни **унікальний** slug (наприклад `break-180-handgun`, `break-180-pcc`, `paper-hits-major-minor-rifle`); **пістолет не виняток**: теж суфікс `-handgun`, щоб уникнути неявного «канону».

**Редакція:** для кожної дисципліни фіксується **`ipsc_edition`** у реєстрі; при оновленні rulebook IPSC — **окремий** рев’ю контенту по дисципліні.

## 7. UX/UI: SOS-плитки та кольори

**Принцип:** великі контрастні зони натискання, читабельність на сонці, керування одним пальцем.

**П’ять категорій і кольорова кодифікація v0:**

| Категорія     | `category`    | Колір (орієнтир) | Примітка        |
|---------------|---------------|------------------|-----------------|
| Safety & DQ   | `safety`      | Червоний         | Найвищий пріоритет уваги |
| Penalties     | `penalties`   | Жовтий / бурштин | Процедури       |
| Scoring       | `scoring`     | Зелений          | Очки, мішені    |
| Equipment     | `equipment`   | **Фіолетовий**   | Додано як 5-й колір (узгоджено з прогалиною «4 кольори / 5 розділів») |
| Match Admin   | `match-admin` | Синій            | MD, перестріли, протести |

Доступність: не покладатися лише на колір — **іконка + текст** категорії.

## 8. Повна структура контенту (категорії та статті v0)

Нижче — **каркас тем** (базові імена slug у колонці «Slug»). Для **кожної** теми випускається **5** статей — по одній на дисципліну `handgun` | `pcc` | `rifle` | `mini_rifle` | `shotgun` — з **унікальним** slug (рекомендовано `{base-slug}-{discipline}`) і відповідним `discipline` + `primary_url`. Кожна така стаття — **пара** Markdown-файлів (`locale: uk` та `locale: en`). Заголовки та тіло EN — англійською; таблиця задає тему українською як **орієнтир для обох** мов. Нумерація `ipsc_refs` у кожній статті звіряється з PDF **своєї** дисципліни (орієнтири в таблицях нижче — від handgun-редакції як приклад для авторів; для інших дисциплін замінити після звірки).

### 8.1. Safety & DQ (`safety`)

| Slug (приклад)        | Тема (UK) | Орієнтир IPSC (якір для рев’ю) |
|-----------------------|-----------|---------------------------------|
| `break-180`           | Розбиття кутів (напрямок ствола за межі дозволеного сектору) | напр. **10.5.2** — звірка з редакцією |
| `accidental-discharge`| Випадковий постріл під час маніпуляцій / руху / близько до інших | напр. **10.4** |
| `trigger-finger`      | Палець у скобі під час перезаряджання, усунення затримки, руху | напр. **10.5.8 / 10.5.9** |
| `dropped-gun`         | Впущена зброя між сигналом таймера та «Range Clear» | напр. **10.5.3** |
| `unsafe-gun-handling` | Небезпечне поводження (виймання без команди, sweeping тощо) | напр. **10.5.1** |

Номери в таблиці — **орієнтир для авторів**; у статті після рев’ю мають відповідати **фактичній** редакції PDF.

### 8.2. Penalties (`penalties`)

| Slug | Тема (UK) |
|------|-----------|
| `foot-fault` | Заступ: 1 процедура за постріл у заступі |
| `significant-advantage` | Значна перевага: 1 процедура за постріл, якщо заступ дав перевагу |
| `failure-to-engage` | Необстріляна мішень (FTE): процедури за мішень |
| `procedural-general` | Порушення процедури / умов брифінгу (загальна логіка) | напр. **10.2.2** після звірки |

### 8.3. Scoring (`scoring`)

| Slug | Тема (UK) |
|------|-----------|
| `paper-hits-major-minor` | Паперові влучання: **Alpha / Charlie / Delta** — обов’язково **дві колонки: Major та Minor** за таблицею обраної редакції IPSC; без публікації цифр без звірки з первинником |
| `steel-poppers` | Метал і поппери: падіння, miss |
| `no-shoots` | Штрафні мішені (No-Shoot): штраф за влучання, обмеження кількості штрафів на мішень |
| `disappearing-targets` | Зникаючі мішені: miss — чи штраф (за правилами матчу / правилами мішені) |

**Відповідь на питання Major/Minor:** у v0 кожна дисципліна має свій slug на кшталт `paper-hits-major-minor-{discipline}`; у статті — **таблиця з офіційного rulebook** цієї дисципліни для обраної редакції (скрин/посилання), а не «загальні» цифри лише для Minor без контексту.

### 8.4. Equipment (`equipment`)

| Slug | Тема (UK) |
|------|-----------|
| `division-overview` | Перевірка дивізіонів і класів — за таблицями **обраної** дисципліни (handgun/PCC/rifle/mini rifle/shotgun різняться) |
| `holster-position` | Положення кобури / носіння (для long gun — еквівалентні вимоги з rulebook цієї дисципліни, не копіювати текст пістолетної кобури) |
| `ready-conditions` | Стартові стани — лише **офіційні** Conditions з PDF поточної дисципліни |

### 8.5. Match Admin (`match-admin`)

| Slug | Тема (UK) |
|------|-----------|
| `reshoots` | Перестріли: коли RO зобов’язаний дати reshoot (несправність обладнання стейджу тощо) |
| `protests-arbitration` | Протести та арбітраж: виклик RM, апеляція |
| `md-ro-roles` | Взаємодія з MD: зони відповідальності (безпека vs логістика) |

## 9. Технічна рамка (імплементація після спеки)

- **Код:** рекомендовано `src/features/ro-helper/` (lazy route); не імпортувати внутрішні модулі **stage-builder** без контракту ([PORTAL_PLAN.md](./PORTAL_PLAN.md) §2.3).
- **Контент:** Markdown; для **офлайн PWA** бажано **імпорт у бандл** (або згенерований індекс при збірці). Файли лише в `public/*.md` з `fetch` **можуть** не потрапити в Workbox precache — див. [TECH.md](./TECH.md) та `vite.config.ts` (`globPatterns`).
- **Рендер:** легкий MD → HTML (бібліотека на вибір на етапі імплементації).

## 10. Quick Cite та авторське право

**Мета:** кнопка копіює в буфер **короткий фрагмент** для звіту / листування.

**Формат v0 (рекомендовано):**

```text
[RO Helper / Shooters Tools]
Тема: {title UK}
IPSC: {пункти} ({ipsc_edition})
Первинник: {primary_url}
Локально (ФПСУ): {якщо є — коротко + URL}
Примітка: не замінює рішення RM / документи матчу.
```

Не копіювати **повні абзаци** офіційного rulebook у буфер за замовчуванням.

## 11. Офлайн (PWA)

## 14. Відповіді на питання з планування (зведено)

| Питання | Відповідь |
|---------|-----------|
| Чи завантажувати всі PDF rulebook у репозиторій? | **Ні**; посилання на офіційні джерела + редакторські статті UK. PDF для **внутрішнього** рев’ю — на диску авторів. |
| Різні види зброї | **П’ять** дисциплін IPSC одразу: `handgun`, `pcc`, `rifle`, `mini_rifle`, `shotgun` — однакова логіка карток; пістолет не «головний» вид. |
| Хто пише Markdown? | Модель **C**: AI-чернетки, **власник** — обов’язкове рев’ю. |
| IPSC vs ФПСУ | IPSC — базис; ФПСУ — **шар** з перемикачем; маппінг **ручний**. |
| 4 чи 5 кольорів у UI? | **П’ять**; Equipment — **фіолетовий**. |
| Major/Minor у скорингу | Окрема стаття з **таблицею з первинника** для обраної редакції; не фіксувати цифри в спеку без редакції. |
| Кілька видів зброї / дисциплін | Окремі **офіційні документи IPSC** на дисципліну; у статті `discipline` + `primary_url` саме на той PDF/HTML. |
| Двомовність | **UK + EN** для кожного `slug`; мова UI порталу вибирає файл; шар ФПСУ незалежний від мови. |
| Хто каже «ФПСУ відрізняється» | **Рев’юер** (обов’язково) за офіційним документом ФПСУ; LLM лише чернетка за витягами; без підтвердження — блок ФПСУ не показуємо. |
| Офлайн | Контент у бандлі / precache; уникати «голого» `fetch` до `public` без стратегії SW. |
| SEO | Клієнтські meta у v0; prerender — пізніше. |
