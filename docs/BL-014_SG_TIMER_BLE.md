# BL-014 — SG Timer: BLE API (референс для інтеграції)

Документ фіксує розбір публічного опису протоколу **SMART SHOT TIMER — BLE API Version 3.2** (файл від виробника, наприклад `sg_timer_public_bt_api-2.pdf`). Оригінал PDF зберігається поза репозиторієм або в архіві команди; у git лишається цей текстовий конспект.

**Пов’язано:** [BACKLOG.md](./BACKLOG.md) **BL-014** (інтеграція з Bluetooth-таймерами).

---

## 1. Підтримувані моделі та ідентифікація в ефірі

Реклама пристрою:

- **Ім’я:** `SG-SST4XYYYYY`, де  
  - **`X`** — модель: **`A`** = SG Timer **Sport**, **`B`** = SG Timer **GO**;  
  - **`YYYYY`** — серійний номер.
- **Сервіс (128-bit UUID):** `7520FFFF-14D2-4CDA-8B6B-697C554C9311` (MAIN Service).

Усі багатобайтові значення в характеристиках — **big-endian**.

---

## 2. Таблиця BLE-характеристик (скорочено)

| Name | UUID (префікс …697C554C9311) | Операції |
|------|------------------------------|----------|
| COMMAND | `75200000-…` | W, N |
| EVENT | `75200001-…` | N |
| SAVED_SESSION_ID_LIST | `75200002-…` | R, W |
| RESERVED | `75200003-…` | R |
| SHOT_LIST | `75200004-…` | R, W |
| PAR_SETUP | `75200005-…` | R, W |
| UNIX_TIME | `75200006-…` | R, W |
| API_VERSION | `7520FFFE-…` | R |

R — read, W — write, N — notify.

---

## 3. Команди та відповіді (COMMAND)

Формат команди: поля `len`, `cmd_id`, `cmd_data` (деталі в PDF). Відповідь приходить через **notification**: `len`, `cmd_id`, `resp_code` (`0x00` success, `0x01` error).

| Command | ID |
|---------|-----|
| SESSION_START | 0x00 |
| SESSION_SUSPEND | 0x01 |
| SESSION_RESUME | 0x02 |
| SESSION_STOP | 0x03 |

Команди можна посилати послідовно без очікування відповіді на кожну; розбір потоку — по байту довжини в початку пакета.

---

## 4. Події (EVENT, notify)

| Event | ID |
|-------|-----|
| SESSION_STARTED | 0x00 |
| SESSION_SUSPENDED | 0x01 |
| SESSION_RESUMED | 0x02 |
| SESSION_STOPPED | 0x03 |
| SHOT_DETECTED | 0x04 |
| SESSION_SET_BEGIN | 0x05 |

**Ключове для імпорту сплітів:**

- **SESSION_STARTED** — `sess_id` (Unix timestamp), `start_delay` (0.1 s).
- **SHOT_DETECTED** (лише в сесії RO) — `sess_id`, `shot_num`, **`shot_time` у мілісекундах**.
- Події зупинки/паузи — також несуть `sess_id` і лічильник пострілів де застосовно (див. PDF).

Ідентифікатор сесії на пристрої — **Unix timestamp** у секундах (узгодження з **UNIX_TIME**).

---

## 5. Дані сесій і пострілів

### 5.1. SAVED_SESSION_ID_LIST

Читання списку збережених сесій: запис `sess_id` (4 байти; `0xFFFFFFFF` — від останньої / найновішої), повторні read для обходу списку; порядок — від новіших до старіших; кінець списку — `0xFFFFFFFF` у відповіді (деталі wraparound у PDF).

### 5.2. SHOT_LIST

Після запису `sess_id` сесії — читання пар **`shot_number`**, **`shot_time`** (час у **1 ms**). Повторні read до кінця списку; кінець — `0xFFFFFFFF` у `shot_time` (див. PDF).

### 5.3. PAR_SETUP (налаштування RO-сесії)

Три поля по 2 байти: **`start_delay`** (0.1 s; `0xFFFF` — випадкова затримка 1.0–4.0 s), **`time_limit`** (0.1 s; `0` — без обмеження часу), **`shot_limit`** (`0` — без обмеження пострілів).

### 5.4. UNIX_TIME

Локальний час пристрою, секунди з Unix epoch — синхронізація часу та узгодження `sess_id`.

### 5.5. API_VERSION

Нетермінований ASCII-рядок версії API у прошивці (наприклад `1.0`); для перевірки сумісності з цим документом (v3.2).

## 6. Що це дає Stage Builder (продукт)

1. **Реальний таймлайн пострілів** у межах сесії: номер пострілу + час у мс, події старту/стопу/паузи — достатньо для **імпорту сплітів** у застосунок.
2. **Параметри вправа (PAR)** з боку таймера — затримка старту, ліміти часу та пострілів; узгоджується з ідеєю структурованої серії пострілів.
3. **Офлайн-історія на пристрої** — можливий імпорт **завершених** сесій без live-з’єднання під час стрільби (після з’єднання з таймером).
4. **Референсний протокол для MVP** — замість абстрактного «формат від виробника» є конкретні UUID, команди та формати полів.

**Що протокол *не* дає:** прив’язку пострілу до **конкретної мішені** на схемі — лише порядок і час; маппінг на таргети сцени лишається **логікою Stage Builder** (користувач, порядок цілей у брифінгу, правила парку тощо).

## 7. Обмеження та ризики

| Тема | Деталі |
|------|--------|
| **Web Bluetooth** | Потрібні **HTTPS**, жест користувача, підтримка браузера (**Chrome/Edge** на десктопі та Android з обмеженнями). **Safari / iOS** — обмежено; для «всіх телефонів» можливий окремий **нативний** шар або інший канал даних. |
| **Один виробник** | Документ описує **SG Timer Sport / GO**; інші таймери матимуть інший BLE-профіль — варто закласти **адаптер** «таймер → внутрішня модель сплітів». |
| **Версія прошивки** | Звіряти **API_VERSION** з очікуваною сумісністю; при оновленні прошивки виробник може змінити деталі. |

## 8. Рекомендації для реалізації (коли дійдуть до коду)

- Ввести **внутрішній формат** «сесія + масив сплітів (час відносно старту або абсолютний)» і один **драйвер SG Timer (BLE)** за цим документом.
- Для MVP обрати сценарій: **live** (EVENT + SHOT_DETECTED) і/або **імпорт** (SAVED_SESSION_ID_LIST + SHOT_LIST).
- Юридично: використання назви та протоколу — за ліцензією/умовами виробника документації; у репозиторії тримати лише **власний** опис полів, не обов’язково PDF.
