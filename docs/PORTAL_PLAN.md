# Портал навколо Stage Builder — план розвитку

Документ для власника продукту та розробки: **архітектурна рамка**, **бренд і URL**, **хостінг**, **модулі**, **дорожня карта** та **freemium** для поступового розширення з **одного репозиторію** (solo / дуже маленька команда). Деталі поточного редактора — у [PRODUCT.md](./PRODUCT.md), [FUNCTIONALITY.md](./FUNCTIONALITY.md), [TECH.md](./TECH.md).

**Статус:** жива стратегія. **Production-домен:** `https://shooters-tools.com` (купівля та DNS через Vercel; після делегування — підключити до проєкту Stage Builder і перевести канонічні URL з коду/env з `stage-builder.vercel.app`).

---

## 1. Мета та контекст

- **Stage Builder** залишається **ядром цінності**: безкоштовний браузерний редактор вправ (2D, 3D, брифінг, PDF, JSON, PWA).
- Навколо нього збирається **модульний портал** (RO Helper, Hit Factor, аналіз мішеней, каталог вправ, профілі тощо) без перетворення кодової бази на «неможливий моноліт»: межі модулів у коді, спільний shell, поступовий бекенд.
- Старт — **безкоштовно**; у перспективі — **freemium / PRO** без раннього відсікання базового сценарію редактора.

---

## 2. Architecture Vision

### 2.1. Підхід

**Рекомендація:** **modular monolith** — один SPA (Vite + React), один деплой, чіткі папки `features/*`, спільний `shared/*` та `core/*` (shell, маршрути, політики).

- **Не на старті:** micro-frontends, окремі репозиторії на кожен модуль, власний Kubernetes.
- **Інтеграція Stage Builder:** окремий **маршрут** (наприклад `/stage-builder`), який монтує поточний редактор; глобальний стан редактора **не** змішувати з іншими модулями без явних контрактів.
- **Стабільні deep links:** шляхи **`/v/:shareId`** (перегляд) та **`/e/:shareId`** (редактор) **не змінювати** — це публічний контракт для змагань і QR.

### 2.2. Шари

| Шар | Відповідальність |
|-----|------------------|
| **Core** | Портальний layout, карта маршрутів, lazy-завантаження, feature flags / env для PRO |
| **Features** | Окремі модулі: `stage-builder`, `ro-helper`, `hit-factor`, … |
| **Shared** | UI-примітиви, токени, i18n-хелпери, analytics-обгортки |
| **Domain** | Логіка вправи та брифінгу залишається переважно в домені Stage Builder; нові домени — окремі піддерева або модулі, щоб уникнути глобальної каші |
| **Data / Auth** | Поступово: один BaaS (наприклад Supabase уже використовується для share) — auth, Postgres, storage |

### 2.3. Правила меж

- Модуль **не** імпортує внутрішні файли іншого feature напряму — лише через **`shared`** або узгоджені **`contracts/*`** (типи DTO, версія JSON вправи).
- Один **router map**; нові модулі = новий lazy-route + запис у короткому **реєстрі модулів** (документ або код).

## 3. Бренд-політика

### 3.1. Ієрархія назв

- **Портал (umbrella):** домен **shooters-tools.com**; у `<title>`, шапці та OG варто дублювати **нішу** (наприклад *IPSC / practical shooting / match tools*), щоб відрізнятися від **іншого** бренду **shooterstools.com** (без дефісу) — магазин gunsmithing-інструментів Borka / Pro-Shot.
- **Stage Builder:** назва **модуля**; у навігації та матеріалах для користувача залишається розпізнаваним брендом редактора.

### 3.2. Юридичні та етичні рамки

- Чіткий **дисклеймер:** продукт **не** є офіційним джерелом правил IPSC чи федерації; не замінює рішення Range Master / арбітражу; на змаганнях керують офіційні документи організатора.
- Цитати з правил — з **посиланням на первинне джерело**; за можливості вказувати редакцію / дату.
- **Торгові знаки:** не заявляти «офіційне партнерство» без угоди; нейтральні формулювання на кшталт «для практичних стрільців / у дусі IPSC» безпечніші для старту.

### 3.3. Тон і візуал

- Один стиль дій у CTA; уникати змішування «розважального» та «юридичного» тону на одному екрані без потреби.
- Єдина палітра та типографіка на весь портал; модулі відрізняються **підзаголовком та іконкою**, а не «іншим сайтом».

### 3.4. Монетизація та довіра

- **Free:** повноцінний базовий сценарій Stage Builder (проєкт → експорт) не обмежувати штучно для adoption.
- **PRO:** прозоро описувати (хмара, квоти, розширений AI тощо), без дрібних обмежень на критичних кроках без пояснення.

## 4. URL та домени

### 4.1. Канонічний хост

- **Зафіксовано:** `https://shooters-tools.com` (apex). Якщо додасте `www` — зробіть **301** на один канон і використовуйте його всюди в мета та посиланнях.
- У продакшені **canonical**, **`og:url`**, посилання в PDF/QR — на **канонічний хост**, а не на `*.vercel.app`, щоб узгодити SEO та бренд у видачі (оновлення в репозиторії: `index.html`, `public/sitemap*.xml`, `public/robots.txt`, `src/lib/resolvePublicOriginFromEnv` / `VITE_SHARE_PUBLIC_ORIGIN` на Vercel, middleware — див. [TECH.md](./TECH.md)).

### 4.2. Структура шляхів (рекомендована)

| Призначення | Шлях | Примітка |
|-------------|------|----------|
| Головна порталу | `/` | Реалізовано: `PortalHome` + картка в Stage Builder |
| Stage Builder | `/stage-builder` | Реалізовано: монтується `App` |
| Інші модулі | `/hit-factor`, `/ro-helper`, … | Стабільні slug |
| Публічна вправа (перегляд) | `/v/:shareId` | **Не ламати** |
| Редактор за посиланням | `/e/:shareId` | **Не ламати** |
| Політика тощо | `/publish-policy` | Як зараз |
| Статичні OG / іконки | `/og-image.png`, favicon | Абсолютні URL на каноні |

### 4.3. Середовища

- **Production:** лише канонічний домен.
- **Preview (Vercel):** окремі URL; бажано **noindex** для не-production (існуючі механізми staging за `VITE_SITE_ENV` зберегти).
- **Staging** окремим піддоменом — опційно, коли з’явиться регулярне QA.

### 4.4. Міграція з `*.vercel.app`

- У Vercel: домен додано до проєкту → **Redirect** з `stage-builder.vercel.app` на `shooters-tools.com` (або залишити обидва з однаковим контентом, але канон — лише новий хост).
- Оновити документацію та приклади URL у [PRODUCT.md](./PRODUCT.md), README та коді після підтвердження DNS.

### 4.5. Чекліст одразу після покупки на Vercel

1. **Connect to project** — прив’язати `shooters-tools.com` до поточного деплою Stage Builder (або нового, якщо розділите репо).
2. Дочекатися **nameserver propagation**; перевірити HTTPS (автоматично на Vercel).
3. У **Environment Variables** production: **`VITE_SHARE_PUBLIC_ORIGIN=https://shooters-tools.com`** (або ваш канон з `www`), щоб share-посилання та OG у Edge відповідали новому хосту.
4. **Google Search Console** — додати property для нового домену, надіслати sitemap після оновлення URL у `public/sitemap*.xml`.
5. Підтвердити **email verification** від ICANN / реєстратора (як у підказці Vercel).

## 5. Хостінг та інфраструктура

### 5.1. Frontend

- **Vercel** (поточний стек): CDN, TLS, preview-деплої, Edge middleware, serverless API для легких ендпоінтів.
- **Custom domain:** налаштування в панелі Vercel + DNS у реєстратора (A/AAAA або CNAME за документацією Vercel).

### 5.2. Backend і дані

- Розширення **Supabase** (уже в проєкті для share): Postgres, auth, storage, Edge Functions — **один основний постачальник** на старті, без розпорошення даних.
- Секрети лише в **змінних середовища** (Vercel / Supabase); ключі AI — **тільки на сервері**.

### 5.3. AI, пошук, важкі задачі

- Легкі виклики: **Vercel Functions / Edge** + rate limiting і кеш.
- Пошук по правилах: спочатку **клієнтський індекс** (наприклад FlexSearch / MiniSearch) над підготовленим JSON; зовнішній search — за потреби обсягу.
- Довгі або дорогі job — **черги / окремий worker** лише коли з’явиться реальне навантаження.

### 5.4. Observability (мінімум)

- **Помилки:** Sentry або аналог, source maps у CI.
- **Продукт:** наявна аналітика + за потреби події «відкрито модуль».
- **Web Vitals** у межах наявного рішення.

## 8. UX Concept (коротко)

- **Home** = launcher задач, не аналітичний dashboard.
- **Навігація:** обмежена кількість пунктів; на mobile — великі зони натискання; **deep links** на `/v/*` ведуть прямо в сценарій перегляду.
- **Desktop:** проєктування вправ і довгі брифінги; **mobile:** калькулятори, шпаргалки, швидкий доступ (PWA).

## 10. Freemium Strategy

- **Завжди free (рекомендація):** повний базовий цикл Stage Builder локально, Hit Factor, базові довідкові матеріали, публічні шаблони з розумним лімітом, read-only за share-посиланням.
- **PRO:** хмарна синхронізація, приватні колекції, розширений AI, додаткові квоти та історія — без блокування «зробив PDF і пішов на змагання».
- **Не ховати за paywall рано:** імпорт/експорт JSON, базовий PDF, публічний перегляд `/v/:id`.

## 11. Ризики та рішення на старті

**Архітектура:** змішування state між модулями; великий початковий бандл без lazy; циклічні імпорти; відсутність версії схеми JSON; секрети на клієнті; неконсистентні URL при появі порталу.

**Продукт:** розмивання фокусу; ранній paywall; завищені очікування від AI; підтримка контенту RO без процесу.

**Рішення зафіксувати заздалегідь:** modular monolith + lazy; канонічний домен і редіректи; не ламати `/v/` та `/e/`; один BaaS для майбутнього auth; `contracts` для JSON; мінімальний design shell; Sentry; політика PWA; контент RO як окремий pipeline (файли / Markdown).
