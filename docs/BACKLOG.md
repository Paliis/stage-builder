# Беклог продукту (ідеї та напрямки)

Єдине місце для **накопичення можливих фіч і змін**, які ще **не** затверджені як поточна робота. Сюди переносимо ідеї з обговорень, [USER_FEEDBACK.md](./USER_FEEDBACK.md) та внутрішнього планування; **періодично** (наприклад, раз на спринт або раз на місяць) — **грумимо**: уточнюємо формулювання, оцінюємо зусилля, **пріоритезуємо**, переносимо в роботу або явно відкладаємо.

**Скрізна нумерація:** усі пункти мають **один** лічильник **`BL-001` … `BL-999`** (префікс **BL** = backlog). Нові ідеї отримують **наступний вільний номер**. Розділи: **активний** беклог, **відкладено**, **архів** (знято з грумінгу, номер і текст зберігаються для історії).

**Не плутати з:** поточний зміст продукту — [FUNCTIONALITY.md](./FUNCTIONALITY.md); реліз-ноти та версії — [VERSIONING.md](./VERSIONING.md).

## Як вести цей файл

| Поле | Зміст |
|------|--------|
| **ID** | **`BL-NNN`** — скрізний номер у всьому беклозі; для посилань у чатах і в USER_FEEDBACK. |
| **Статус** | `idea` — сирий запис; `candidate` — зрозуміло «що» і «навіщо»; `ready` — можна брати в спринт після рішення; `in progress` — в роботі; `done` — зроблено; `dropped` — свідомо не робимо / застаріло. |
| **Пріоритет** | `P0` … `P3` або `—` поки не голосували. Після грумінгу оновлювати. |
| **Джерело** | Напр. `USER_FEEDBACK`, обговорення 2026-04-08, внутрішнє. |

Після грумінгу варто коротко зафіксувати рішення в колонці **Примітки** (наприклад: «взяти в Q2», «зачекати на бекенд», «об’єднати з …»).

### Покроковий процес (без фантазій)

1. **Задачі в беклозі** — формулювання, ID, дата додавання (нижче).
2. **Оцінка** — окремий крок: T-shirt (S/M/L/XL) або людино-години; заповнюється колонка **Оцінка** після обговорення.
3. **Пріоритезація** — після оцінки: P0…P3 або порядок у шарі.
4. **Архітектура та техстек** — лише для обраних до реалізації пунктів; окремий документ або підрозділ після п. 3.

Якщо для пункту **не вистачає вимог** — не оцінюємо до уточнення; питання — у колонці **Питання / ризики**.

### BL-010 — специфікація (уточнення)

*Відмінність від поточного продукту:* зараз — один сектор безпеки від **стартової позиції** за кутами з брифінгу ([FUNCTIONALITY.md](./FUNCTIONALITY.md)). **BL-010** — **глибша логіка**: умовно для **кожної точки** простору, що відповідає зоні стрільби **нижче крайніх точок штрафних ліній** (тобто в допустимій області, обмеженій лініями), аналізувати:

1. Чи є **видимі мішені за межами кута безпеки 180°** (відносно напрямку стрільби / локальної орієнтації з цієї точки) — тобто «простріл» поза безпечним півколом.
2. З яких із цих точок мішень **не прикрита декораціями** (реквізит як перешкода для променя з точки до мішені) — щоб відрізнити «теоретично в секторі» від «реально прострелюється» з урахуванням щитів, бочок тощо.

*Реалізація:* окремий обчислювальний шар (дискретна сітка або вибірка точок уздовж межі зони, промені / кути на 2D-плані; за потреби — спрощені профілі висоти). Це **не** дублікат одного клину від старту, а **глобальніший** аналіз сцени.

**Відкрито для наступного грумінгу:** крок дискретизації точок; які типи `props` вважати непрозорими для променя; чи достатньо 2D-проєкції для першого MVP.

### BL-010 та BL-013 — спільна логіка (геометрія видимості)

**BL-010** (Community, безпека) і **BL-013** (PRO, дизайн) **перетинаються**: обидва потребують **променів / видимості** з точки спостереження до мішені з урахуванням **перешкод** (реквізит як декор / укриття) на 2D-плані (і за потреби спрощених припущень по висоті).

| Аспект | BL-010 | BL-013 |
|--------|--------|--------|
| **Мета продукту** | Попередження про небезпечні простріли (кут 180°, зона штрафних ліній) | Підказки дизайнеру: видимі зони, «зайві» простріли |
| **Спільне ядро** | Те саме: обчислення перетину променя з полігонами перешкод, список «хто перекриває» лінію огляду | Те саме |
| **Відмінність** | Правила безпеки, дискретизація точок уздовж допустимої зони, агрегація попереджень | Візуалізація зон, можливо інші точки огляду (ручний вибір) |

**Рекомендація для планування:** одна **технічна історія** на спільний модуль (наприклад `visibility` / `raycast2d` у `src/domain/`), потім **два продуктові шари** — спочатку MVP ядра, потім BL-010 і/або BL-013. Оцінку давати **пакетом** або явно ділити: «ядро + інтеграція BL-010» / «ядро + інтеграція BL-013», щоб не подвоювати зусилля.

### Короткий аналіз (думка для обговорення)

- **Шар 1** найближче до поточного Stage Builder: **BL-001** уже покриває «поділитися 3D»; **BL-009** (генератор) реалістичний після чітких правил; **BL-010** + **BL-013** — **узгодити спільне ядро** (див. блок вище), потім окремі оцінки інтеграції; **BL-008** (AR) — окремий великий продуктовий ризик, не «фіча в спринті» без MVP-скопу.
- **Шар 2** вимагає **даних** (HF, таймери) і **інтеграцій**; без вибору джерела даних і одного референсного таймера оцінки будуть умовними.
- **Шар 3** — **екосистема**: платежі, контент, довіра; логічно після стабільного шарингу та (за потреби) акаунтів.
