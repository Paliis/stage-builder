# ТЗ: підготовка статей RO Helper (v0)

**Призначення:** операційне завдання для виробництва Markdown-статей модуля **RO Helper** за специфікацією [RO_HELPER_V0.md](./RO_HELPER_V0.md). Пайплайн: **модель C** — чернетка з LLM, **обов’язкове рев’ю** людиною перед публікацією.

**Останнє оновлення:** квітень 2026 (UK+EN, дисципліни, ФПСУ — хто верифікує; пакет для рев’ю ZIP).

**Мапінг розділів IPSC / ФПСУ на категорії карток:** [RO_HELPER_IPSC_FPSU_TOPIC_MAP.md](./RO_HELPER_IPSC_FPSU_TOPIC_MAP.md).

**Словник термінів IPSC (EN) ↔ ФПСУ (UK) для чернеток карток:** [RO_HELPER_IPSC_FPSU_GLOSSARY.md](./RO_HELPER_IPSC_FPSU_GLOSSARY.md) — **чернетка; потребує верифікації** рев’юером з первинниками.

**Повна матриця карток:** [RO_HELPER_CARD_MATRIX.md](./RO_HELPER_CARD_MATRIX.md) ([CSV](./RO_HELPER_CARD_MATRIX.csv)).

## 1. Результат роботи (deliverables)

1. Для **кожної** картки з [RO_HELPER_CARD_REGISTRY.md](./RO_HELPER_CARD_REGISTRY.md) (або узгодженого доповнення до неї) — **два** файли **`*.md`** (`locale: uk` та `locale: en`) з валідним **YAML frontmatter** ([§2.2](./RO_HELPER_V0.md#22-поля-контенту-рекомендований-frontmatter)): однакові `slug`, `discipline`, `category`, `card_id` (якщо призначено), `ipsc_refs`, узгоджені `primary_url` (зазвичай однакові для пари мов). Одна стаття = **`slug` + `discipline`**; див. також [§5](./RO_HELPER_V0.md#5-інформаційна-архітектура-та-url).
2. Звірені пункти IPSC з **правильного** PDF для **`discipline`** (завжди той rulebook, який відповідає полю `discipline`; не змішувати документи).
3. Блок **«Локально (ФПСУ)»** — лише якщо виконано [RO_HELPER_V0.md §4.1](./RO_HELPER_V0.md#41-хто-фіксує-відмінності-фпсу-від-ipsc); інакше розділ відсутній, `fpsu_delta_verified: false`.
4. Рядок у **`content/ro-helper/INDEX.md`**: slug, `discipline`, статуси **uk** / **en** окремо, `fpsu_delta_verified`, рев’юер, дати.

До старту пакета зафіксувати **`ipsc_edition`** **окремо для кожної** з п’яти дисциплін (у реєстрі може бути блок або окремі колонки / рядки на discipline).

### 1.0. Вивантаження пакета для перевірки (рев’ю офлайн)

Щоб надіслати рев’юеру **усі наявні Markdown-картки** та контекст матриці одним архівом:

```bash
npm run ro-helper:export-review
```

Перевірка наявності статей за матрицею, полів frontmatter (`ipsc_refs`, `fpsu_refs` з URL на upsf.org.ua), секцій **IPSC** / **Локально (ФПСУ)** та попереджень якості (наприклад UK «дуло» за §1.1):

```bash
npm run ro-helper:validate
```

Команда **`npm run check`** (lint + тести + збірка) також запускає **`ro-helper:validate`**.

Вигрузка для **перевірки релевантності** (таблиця + JSON + інструкція форми + контекст матриць у ZIP):

```bash
npm run ro-helper:export-relevance
```

У **`exports/`** з’явиться **`ro-helper-relevance-<timestamp>.zip`**: `relevance.csv` (колонки `manual_*` для ручного заповнення), `relevance.json`, `REVIEW_FORM_UA.md`, `schema.json`, за наявності — `context/*.csv`. Деталі колонок — у `REVIEW_FORM_UA.md` всередині архіву.

Промпт для **другого агента** (перехресна перевірка покриття, якості та IPSC/ФПСУ): [RO_HELPER_CROSS_REVIEW_AGENT_PROMPT.md](./RO_HELPER_CROSS_REVIEW_AGENT_PROMPT.md).

У каталозі **`exports/`** (не комітиться в git) з’явиться файл на кшталт **`ro-helper-review-<timestamp>.zip`**. Усередині:

- `content/ro-helper/**` — статті UK/EN та `INDEX.md`;
- `docs/RO_HELPER_CARD_MATRIX.csv` та `docs/RO_HELPER_CARD_MATRIX.md` — повна матриця;
- `READ_ME_UA.txt` — короткі інструкції.

**Не входить:** inline-демо з `src/portal/roHelperCardDemoModel.ts` (окремий шар прототипу); за потреби рев’ю демо — дублювати у `content/ro-helper/` або описати в листі до рев’юера.

### 1.1. Українська термінологія (узгоджено з офіційним стилем ФПСУ)

У текстах **`locale: uk`** для RO Helper:

- **Не** використовувати розмовне слово **«дуло»** у значенні, яке в офіційних документах ФПСУ послідовно передають термінами **ствол** або **дульний зріз** (залежно від контексту).
- **Контекст безпеки та кутів (muzzle direction, 180°, sweeping):** формулювати напрям як **«ствол зброї спрямований…»** / **«напрямок стволу»**; так само в наступних картках цього типу.
- **Дульний зріз** доречний, коли акцент на **площині вильоту** / точці відліку траєкторії; при сумніві між «ствол» і «дульний зріз» погоджувати з рев’юером за текстом ФПСУ.
- У **англомовних** статтях (`locale: en`) залишається канонічне **muzzle** з первинника IPSC; відмінність лише в українському шарі формулювань.

## 4. Порядок виконання (по одній статті)

### Крок A — Підготовка контексту

1. Визначити **`discipline`** статті й відкрити відповідний **PDF IPSC**; не змішувати номери пунктів між дисциплінами.
2. Знайти розділ за **`ipsc_refs`** з [§8](./RO_HELPER_V0.md#8-повна-структура-контенту-категорії-та-статті-v0); зафіксувати **`ipsc_edition`**.
3. **ФПСУ:** якщо плануєте блок «Локально» — відкрити **релевантний** документ ФПСУ і **вручну** відмітити, чи є відмінність; інакше блок не додавати.

### Крок B — Чернетка (LLM)

Згенерувати **дві** версії (спочатку UK, потім EN на основі узгодженого UK, або два окремі виклики). **Промпт-рамка:**

```text
Ти допомагаєш підготувати статтю для RO Helper (Shooters Tools).
Мова статті: {українська АБО англійська — одна на файл}.
Дисципліна: {discipline — одна з handgun|pcc|rifle|mini_rifle|shotgun}. PDF IPSC для цієї дисципліни — єдине джерело номерів пунктів.
Не вигадуй номери. Орієнтири пунктів: {ipsc_refs з RO_HELPER_V0 §8 для slug}.
Редакція: {ipsc_edition}.

Завдання:
1) Заголовок і перший розділ (UK: «Що це» / EN: «What this is») — 2–4 речення.
2) Дії RO (UK: «Що робить RO» / EN: «RO actions»).
3) Розділ IPSC — пункти + primary_url (плейсхолдер замінити на офіційне посилання).
4) «Локально (ФПСУ)» / «Local (FPSU)» — ТІЛЬКИ якщо в контексті надано підтверджені витяги з документа ФПСУ; інакше явно: без цього розділу.
5) Типові помилки (1–3 пункти).
Не цитуй довгі абзаци rulebook дослівно.

Slug: {slug}
category: {category}
```

Після відповіді — **frontmatter** вручну; для EN-файлу `locale: en`, `title` англійською.

### Крок C — Рев’ю (людина, обов’язково)

Пройти [чекліст RO_HELPER_V0 §2.1](./RO_HELPER_V0.md#21-чекліст-ревю-перед-релізом-статті) **для UK і для EN** окремо:

- [ ] Пункти IPSC звірено з PDF **потрібної дисципліни**.
- [ ] UK і EN **узгоджені** за змістом (ті самі правила, та сама логіка ФПСУ).
- [ ] Блок ФПСУ: лише якщо рев’юер підтвердив за документом; тоді `fpsu_delta_verified: true`.
- [ ] `status: reviewed`, `reviewer`, `review_date` в **обох** файлах.

### Крок D — Публікація

`published` лише коли **обидві** мови у статусі `reviewed` (або вище) і модуль готовий їх показати.

---

## 5. Структура тіла статті

**UK-файл:** `## Що це`, `## Що робить RO`, `## IPSC`, опційно `## Локально (ФПСУ)`, `## Типові помилки`.

**EN-файл:** ті самі змістові блоки англійською, наприклад `## What this is`, `## RO actions`, `## IPSC`, опційно `## Local (FPSU)`, `## Common mistakes`.

Мінімальна довжина: **орієнтир 300–800 слів** на файл.

---

## 6. Критерії приймання пари статей

Пара **UK + EN** для одного `slug` вважається готовою до `published`, якщо:

1. У кожному файлі заповнені обов’язкові поля з [§2.2](./RO_HELPER_V0.md#22-поля-контенту-рекомендований-frontmatter), включно з `discipline`, коректним `locale`.
2. `slug` і `discipline` збігаються між файлами.
3. Для **scoring** — числа лише з первинника для цієї дисципліни.
4. Заголовки розділів **однією мовою** в межах файлу (UK або EN).

## 7. Пріоритет черги (рекомендований порядок написання)

У межах кожного пункту — послідовно по **дисциплінах** (як у реєстрі) або спочатку завершити категорію для однієї дисципліни, за домовленістю команди.

1. **safety** — усі теми з [§8.1](./RO_HELPER_V0.md#81-safety--dq-safety) × 5 дисциплін.
2. **penalties** — §8.2 × 5.
3. **scoring** — §8.3 × 5 (особлива увага Major/Minor у кожному PDF).
4. **equipment** — §8.4 × 5.
5. **match-admin** — §8.5 × 5.

## 9. Зв’язок з розробкою

Після набору статей у `reviewed` розробник:

- підключає шляхи до файлів або імпорт у бандл (див. [RO_HELPER_V0.md §9](./RO_HELPER_V0.md#9-технічна-рамка-імплементація-після-спеки));
- не змінює **`slug`** без редіректу / міграції.

Питання по UX (Quick Cite, шар ФПСУ) — лише з [RO_HELPER_V0.md](./RO_HELPER_V0.md).

---

## 10. Типові помилки (не робити)

- Публікувати **лише UK** без EN (або навпаки) у фінальному релізі пари.
- Плутати **PDF handgun** з rifle/shotgun при номерах пунктів.
- Публікувати текст моделі **без** перевірки PDF.
- Копіювати **цілі сторінки** rulebook у статтю.
- Змінювати slug «для краси» після того, як хтось уже послався на URL.
- Ставити конкретні очки Charlie/Delta **лише для Minor**, не згадуючи Major (для handgun).
- Писати блок ФПСУ **без** перевірки рев’юером за офіційним документом ФПСУ (або покладатися на «модель каже, що відрізняється»).
- У **`locale: uk`** уникати розмовного **«дуло»** там, де за [§1.1](#11-українська-термінологія-узгоджено-з-офіційним-стилем-фпсу) потрібні **ствол** / **дульний зріз** (безпека, кути).

## 11. Хто порівнює правила ФПСУ з IPSC

Коротко: **відповідальний рев’юер** (ви або делегат) **сам** звіряє офіційні тексти. LLM може прискорити **чернетку**, але не є джерелом істини. Деталі — [RO_HELPER_V0.md §4.1](./RO_HELPER_V0.md#41-хто-фіксує-відмінності-фпсу-від-ipsc).
