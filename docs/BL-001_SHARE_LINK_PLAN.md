# BL-001 — план дій: посилання на вправу (share link)

Короткий URL замість (або поруч із) завантаження `*.stage.json`. **Статус:** план реалізації; після затвердження оновлювати цей файл фазами.

**Пов’язано:** [BACKLOG.md](./BACKLOG.md) **BL-001**; деплой — Vercel (`vercel.json`); формат сцени — `stageProjectFile.ts` / `*.stage.json`.

## Прийняті рішення (продукт)

*Оновлювати при зміні політики.*

| Тема | Рішення |
|------|---------|
| **Логін** | Поки **без логіну**; створення посилань **анонімно**. |
| **Модель довіри** | **Прийнятна:** хто має URL — може відкрити вправу; список «мої посилання» **не** в MVP. |
| **Квота створення** | **50 нових посилань на добу** на одного клієнта (для аноніму — орієнтир: **per IP**, уточнити в реалізації з урахуванням NAT/IPv6). |
| **Перевищення квоти** | Показувати **капчу** (наприклад Cloudflare Turnstile або hCaptcha) і після успіху дозволяти подальше створення в межах розумного; деталі інтеграції — у задачі на бекенд. |
| **Термін життя посилання** | За замовчуванням **365 днів** з моменту створення (`expires_at`); безстроковість і продовження — не в MVP, за потреби пізніше. |
| **Режим «перегляд» (стрілки)** | Лише **перегляд**, **без редагування** сцени: доступні **2D**, **3D** і **брифінг** для читання (камера/масштаб — так; зміна мішеней, поля, брифінгу — **ні**). |
| **Режим «редактор»** | Повноцінне редагування, як після відкриття файлу (за домовленістю з чернеткою — див. нижче). |
| **Один URL — один режим** | Режим задається **явним префіксом шляху** в URL (див. нижче), **без** прихованого перемикання по query, щоб не переплутати посилання в чатах. |
| **Два посилання на одну вправу** | **Допустимо** (і очікувано): окреме посилання для **стрілків** (перегляд) і окреме для **редакторів/організаторів** (редактор). У БД — два записи з полем `mode` / дві ланки на той самий payload (деталі в реалізації). |
| **Префікси URL (приклад)** | **`/v/:shareId`** — тільки перегляд; **`/e/:shareId`** — редактор. Префікси **`v`** / **`e`** обов’язково **різні**; точний кореневий шлях (`/share/v/...` тощо) узгодити при імплементації та **rewrites** на Vercel. |
| **Чернетка й редактор** | Якщо користувач уже в режимі перегляду й потрібен сценарій «відкрити як у редакторі» — відкривати **у новій вкладці** (`target=_blank`), щоб не змішувати перегляд і редагування в одній вкладці. |
| **Публікація = новий id** | Кожна **нова публікація** в хмару отримує **новий** `shareId` і тим самим **новий URL**; **перезапису** вмісту за тим самим посиланням **немає** в MVP (старі посилання вказують на знімок на момент публікації). |
| **Таймстамп у URL (опційно)** | Унікальність і зрозумілість: **можна** закладати **timestamp** у частину шляху або в сам `shareId` (наприклад префікс дати + випадковий суфікс) — деталі генерації id в реалізації; вимога продукту — **завжди новий** ідентифікатор при публікації. |
| **Видалення з сервера** | Поки **не** надаємо можливості видалити опубліковану вправу (немає UI й API видалення в MVP). |
| **Приватність (MVP)** | Усе **публічно** для тих, хто має URL — **не** додаємо паролі, приватні режими та зайву складність. |
| **Політика та згода** | Окремий текст **політики публікації** — **[PUBLISH_POLICY.md](./PUBLISH_POLICY.md)**. Перед публікацією — чекбокс (або еквівалент) і фраза на кшталт: *«Публікуючи, ви погоджуєтесь з [політикою публікації]»* (посилання на сторінку/модалку з текстом). |
| **Скарги** | У політиці зазначено: звернення через існуючий блок **«Зворотний зв’язок»** у **футері** (Email / Telegram), з вказанням URL вправи. Окремий канал скарг **не** вводимо. |
| **Файл і посилання** | **Обидві опції лишаються:** завантаження **`*.stage.json`** і **«Поділитися»** (хмара) — **поруч**, без заміни одного іншим. |
| **Чернетка vs відкриття посилання** | Якщо у користувача вже є **чернетка** в `localStorage`, а він відкриває вправу **за share-URL** — показати **діалог**: зберегти / відкинути попередню чернетку / скасувати відкриття (не перезаписувати мовчки). |
| **QR у PDF** | У QR-коді в брифінгу (експорт PDF) кодувати **посилання на вправу** (share URL), а не лише головну сторінку застосунку. Типово — посилання в режимі **перегляду** (`/v/...`), щоб стрілки одразу бачили сцену; якщо зручніше для організаторів — узгодити в UI (наприклад два QR або вибір). |
| **Мова в URL** | На **момент формування** посилання (копіювання / QR / API) підставляти **мову з `localStorage`** (як у застосунку): наприклад query **`?lang=uk`** / **`?lang=en`**, щоб відкриття лінка відкривало UI тією ж мовою, якою користувач ділився. |
| **Короткий домен** | Використовувати **короткий домен** для share-URL (зручність у чатах і на друку); технічно — **редирект** на основний хост або окремий **Vercel-проєкт / DNS** з тим самим SPA. Конкретне ім’я домену — змінна оточення (наприклад `VITE_SHARE_PUBLIC_ORIGIN`); узгодити при деплої. |
| **Open Graph (OG)** | Для прев’ю в месенджерах і соцмережах: **`og:title`** = **назва вправи** зі сцени (`stage.name` / аналог у JSON); за потреби короткий опис. Реалізація: **динамічні** мета-теги потребують **SSR**, **Edge** з HTML для ботів або **окремого** endpoint, що повертає сторінку з мета — вибір у задачі на імплементацію; дані назви зберігати в БД поруч із payload. |

**Відкрито для реалізації:** відновлення доступу до редагування без акаунта (edit token у відповіді POST — збереження на стороні клієнта).

## 0. Рішення продукту (закрити до коду)

Нижче — **розгорнутий перелік питань**, на які варто дати явні відповіді (навіть якщо «поки ні» або «потім»). Частину вже закрито в таблиці **«Прийняті рішення»** вище.

### 0.1. Хто і як створює посилання

- **Ідентифікація автора:** ~~чи достатньо анонімного створення~~ → **так, без логіну (MVP).** Акаунти — пізніше, якщо знадобиться.
- **Якщо анонімно:** **модель «хто знає посилання — бачить вправу»** прийнята; списку «мої посилання» немає — як користувачу **відновити** доступ до редагування, якщо загубив **edit token** — окреме питання (покластися на збережений токен / нове посилання).
- **Квоти:** **50 / доба**, перевищення → **капча** (див. таблицю вище).
- **Термін життя:** **365 днів** за замовчуванням (див. таблицю вище).

### 0.2. Що саме бачить отримувач посилання (режими)

- **Перегляд (стрілки):** **лише перегляд**, без редагування — **2D**, **3D**, **брифінг** (читання). Рішення — у таблиці **«Прийняті рішення»**.
- **Редактор:** окремий тип посилання / префікс **`/e/`** — повний редактор; **два посилання** на одну вправу (стрілки + редактори) — **допустимо**.
- **URL:** один URL = один режим; **явні префікси** **`/v/`** та **`/e/`** — див. таблицю вище.
- **Перехід у редактор з перегляду:** **нова вкладка** (не модалка в тій самій вкладці) — див. таблицю вище.

### 0.3. Оновлення та версії на сервері

- **Новий id при кожній публікації:** **так** — кожне «опублікувати» створює **новий** запис і **новий URL**; попередні посилання лишаються на **старих** знімках. Пояснення в UI: короткий текст на кшталт «кожна публікація дає нове посилання; старі посилання не змінюються».
- **Таймстамп у URL:** **опційно** (див. таблицю **«Прийняті рішення»**) — для унікальності/орієнтації в часі; не обов’язково окремий сегмент, якщо id уже унікальний.
- **Історія версій на сервері:** не вимагаємо окремої «стрічки версій» у MVP — лише **окремі** записи на кожну публікацію (фактично історія через множинні посилання, якщо користувач їх зберіг).
- **Видалення:** у MVP **немає** — див. таблицю вище; пізніше — за наявності акаунтів або токена власника.

### 0.4. Приватність, контент і відповідальність

- **Публічність:** у MVP **усі** опубліковані посилання — **публічні** для тих, хто має URL; додаткових режимів приватності **немає** (див. таблицю **«Прийняті рішення»**).
- **Індексація пошуковиками:** для сторінок **`/v/`** / **`/e/`** зазвичай бажано **`noindex`** (щоб не роздувати індекс тимчасовими посиланнями) — уточнити в реалізації SEO.
- **Пароль / приватні посилання:** **не** в MVP.
- **Політика + згода:** текст — **[PUBLISH_POLICY.md](./PUBLISH_POLICY.md)**; у UI — чекбокс і фраза про погодження перед публікацією.
- **Рекомендація щодо ПІБ:** у політиці — коротко нагадати не вносити зайві персональні дані в брифінг; окремий розгорнутий дисклеймер не ускладнюємо.
- **Скарги:** через **зворотний зв’язок у футері** (Email / Telegram), як описано в політиці; окремий канал не потрібен.

### 0.5. Інтеграція з поточним продуктом

- **Файл vs посилання:** **обидва** варіанти залишаються — «Завантажити JSON» і «Поділитися» (див. таблицю **«Прийняті рішення»**).
- **Чернетка в `localStorage`:** при відкритті вправи за share-URL, якщо чернетка вже є — **діалог** (зберегти / скасувати / відкинути чернетку), без мовчазного перезапису.
- **QR у PDF:** **посилання на вправу** (share), типово режим перегляду — див. таблицю вище; реалізація в `exportBriefingPdf` / джерелі URL після появи share.

### 0.6. Мова, бренд, домен

- **Мова:** при **формуванні** URL — брати мову з **`localStorage`** і додавати **`?lang=`** (див. таблицю **«Прийняті рішення»**).
- **Короткий домен:** **так** — share-посилання на короткому домені; деталі DNS / env — при імплементації.
- **OG:** **назва вправи** в `og:title` (і за потреби опис); технічно — SSR / Edge / бот-сторінка (див. таблицю вище).

## 1. Архітектура даних (Supabase або аналог)

| Елемент | Опис |
|---------|------|
| **Таблиця `shared_stages`** (назва умовна) | `id` (short public id), **`mode`**: `'view' \| 'edit'` (префікси **`/v/`** / **`/e/`**), `payload` (JSONB), **`title`** (рядок — назва вправи для OG / прев’ю, дубль з `stage.name` при збереженні), **`locale`** (опційно — `uk` / `en` на момент публікації для `?lang=`), `created_at`, `expires_at`, `edit_token_hash` (nullable), `schema_version`; *опційно* `share_group_id` |
| **RLS** | Політики: `INSERT` для anon (з обмеженнями), `SELECT` за `id`, `UPDATE` лише з валідним edit token (через Edge Function, не прямий anon update) |
| **Розмір** | Обмежити розмір JSON (наприклад ≤ 512 KB) на рівні API |

Альтернатива без власного API: **Supabase Storage** (один JSON на об’єкт) + метадані в таблиці — зручно для великих файлів; для MVP часто достатньо **одного JSONB** у Postgres.

## 2. Backend / API (мінімальний контур)

1. **POST `/api/share`** (Vercel Serverless або Supabase Edge Function)
   - Body: JSON сцени (той самий формат, що `*.stage.json`).
   - Відповідь: `{ id, url, editToken? }` — `editToken` показувати **один раз**, якщо плануєте оновлення.
2. **GET `/api/share/:id`** (або лише клієнт через Supabase client `select`)
   - Повертає JSON для гідрації стору.
3. **PATCH** — у MVP **не** потрібен (немає перезапису за тим самим id); оновлена сцена = **нова POST** → новий `shareId`.
4. **Rate limiting** — Vercel Edge Middleware або ліміт у функції (per IP + sliding window).

Якщо використовуєте **лише Supabase клієнт з ключа anon** у браузері — RLS має бути безпечним; **service role** — лише на сервері, ніколи в клієнті.

## 3. Фронтенд (React SPA)

1. **Маршрутизація**
   - Маршрути **`/v/:shareId`** (перегляд) та **`/e/:shareId`** (редактор) — префікси **не плутати** (див. § «Прийняті рішення»).
   - **Vercel:** `rewrites` для SPA — усі `/*` → `index.html` (якщо ще не налаштовано автоматично для Vite).

2. **Потік «Відкрити за посиланням»**
   - **`/v/:shareId`:** `fetch` → `parse` → гідратація; **read-only** для сцени та брифінгу; дозволені лише дії перегляду (2D/3D навігація). Без збереження на сервер.
   - **`/e/:shareId`:** як відкриття файлу — повний редактор; перехід **з** вкладки перегляду **в** редактор — **нова вкладка** (див. `target=_blank` при «Відкрити в редакторі»).
   - **Конфлікт чернетки:** якщо вже є сесійна чернетка — **діалог** перед підміною стану (див. §0.5).

3. **Потік «Поділитися»**
   - Кнопка(и) поруч із Save/Open: «Посилання для стрілків» / «Посилання для редакторів» → POST з полем **`mode`** → повернути **два URL** або два окремі запити — узгодити в UI; копіювання + опційно QR.
   - **PDF:** QR у брифінгу — **URL share** (див. §0.5); `exportBriefingPdf` / `APP_URL` замінити або доповнити параметром, коли share існує.

4. **Змінні середовища**
   - `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (для клієнта); секрети — у Vercel env для функцій.

## 7. Додаткові відкриті питання (техніка та експлуатація)

### 7.1. Інфраструктура

- **Постачальник БД:** чи остаточно **Supabase** (Postgres + Auth + RLS), чи розглядаємо **Vercel Postgres + Serverless** без Supabase? Критерії: вартість на старті, ліміти безкоштовного рівня, зручність RLS.
- **Регіон даних:** чи є вимога зберігати дані **в ЄС** (GDPR) для частини користувачів — чи достатньо вибору регіону Supabase при створенні проєкту?
- **Резервне копіювання:** чи потрібен **експорт** усіх опублікованих вправ для власника (на випадок міграції з платформи)?

### 7.2. API та контракт

- **Версія формату файлу:** чи при збереженні в хмару **завжди** нормалізуємо до останньої `STAGE_PROJECT_VERSION`, чи приймаємо старі версії як при відкритті файлу?
- **Частковий payload:** чи можна зберігати лише **частину** полів (наприклад без чернетки брифінгу) — щоб зменшити розмір — чи завжди **повний** знімок як у `*.stage.json`?
- **Ідемпотентність POST:** чи потрібен **клієнтський** ключ запиту (request id) для повтору при обриві мережі, щоб не створити дублікати посилань?

### 7.3. Спостережуваність

- **Аналітика переходів:** чи збираємо лише **агрегувати** (кількість відкриттів по `id` без fingerprint), чи достатньо подій у **GA4** / Vercel Analytics на клієнті?
- **Логи сервера:** на скільки днів зберігати логи помилок API (rate limit, 413 payload too large)?

### 7.4. Доступність і офлайн

- **PWA:** чи після відкриття посилання вправу можна **відкрити офлайн** з кешу (як зараз локальний файл), чи share — **завжди онлайн**?
- **Часовий зсув:** чи важливо показувати **час створення** посилання в UI для організатора?

### 7.5. Юридичне та підтримка

- **Умови використання:** чи потрібно оновити **Terms** / Privacy Policy щодо хостингу контенту користувачів та посилань?
- **DMCA / видалення:** чи описуємо процедуру **видалення** вправи за зверненням (email шаблон)?

### 7.6. Промо та якість

- **Брендований короткий домен** (окремий DNS / redirect на основний домен) — чи це must-have для маркетингу чи можна відкласти?
- **Прев’ю в месенджерах:** чи потрібні **og:image** з мініатюрою плану (складно без SSR) — чи достатньо текстового заголовка вправи?
